SLIDE 3: The Problem Space (Why SmartSec?)
--------------------------------------------------------------------------------
[Slide Title]: Problem Statement: Vulnerabilities in Current Defenses
[Slide Bullets]:
  * The Zero-Day Gap: Modern malware and sweeps bypass signature databases.
  * Alert Fatigue: Floods of raw firewall logs exhaust security admins.
  * Isolated Threat Intelligence: Phishing checkers, IP lists, and scanners 
    operate in silos, slowing down human response times.
  * Static Risk Assessment: Threat models do not factor in time, causing
    outdated incidents to inflate the current risk landscape forever.

[Visual Recommendations]:
  * An arrow diagram representing the flow of an attack bypassing a standard
    static firewall but being blocked by SmartSec's anomaly detection.

[Speaker Notes]:
  "We identified four critical vulnerabilities in contemporary corporate defenses.
   First, the zero-day gap. Second, alert fatigue—administrators are bombarded 
   with thousands of non-contextual alerts. Third, isolated threat intelligence—
   scanners check URLs but don't connect that data back to active system risks. 
   And fourth, static risk models—if a user fails a login on Monday, their risk 
   profile shouldn't remain critically high three weeks later if no other 
   incidents occur. SmartSec was built from the ground up to solve these four 
   problems."

SLIDE 5: Module 1 - AI-Powered Intrusion Detection (IDS)
--------------------------------------------------------------------------------
[Slide Title]: Module 1: Unsupervised Network Anomaly Detection
[Slide Bullets]:
  * The Algorithm: Isolation Forest (Scikit-Learn).
  * Why Isolation Forest? It isolates anomalies directly instead of profiling 
    normal data, making it highly efficient on high-dimensional traffic.
  * Feature Vectors: Analyzes packet sizes, request intervals, protocol flags,
    and port sweeps.
  * Synthetic Traffic Simulator: Built-in backend service simulating both standard
    operational traffic and live network port-scans to test detection flags.

[Visual Recommendations]:
  * A scatter plot or decision-tree graphic illustrating how Isolation Forest 
    randomly splits partitions to isolate outliers in fewer steps.

[Speaker Notes]:
  "Let's dive into Module 1: the Intrusion Detection System. Instead of using 
   standard classification algorithms that require labeled historical data, we 
   implemented an unsupervised 'Isolation Forest'. This algorithm works by 
   building random decision trees. Because anomalous data points require very few 
   splits to isolate compared to normal dense clusters, the tree isolates them 
   quickly near the root. We also built an internal traffic simulator that spits 
   out port scans and sweeps to prove the algorithm flags them with near-zero latency."

SLIDE 6: Module 2 - Advanced Phishing & Threat Scanner
--------------------------------------------------------------------------------
[Slide Title]: Module 2: Aggregated Multi-API Threat Scanner
[Slide Bullets]:

  * Phase 2: Parallel Third-Party Threat Intelligence Calls:

    - VirusTotal v3 API (Queries multi-engine hash/domain antivirus results)
    - AbuseIPDB API (Verifies server/IP reporting history)
  * Asynchronous Performance: Python 'asyncio' fires APIs concurrently to
    prevent frontend dashboard blockages.

[Visual Recommendations]:
  * A step-by-step scanner workflow:
    [URL Input] -> [19 Heuristic Tests] -> [Asynchronous API Calls] -> [Verdict]

[Speaker Notes]:
  "Module 2 is our Multi-API Threat Scanner. When a user input-scans a URL, the 
   system first runs 19 static heuristic checks locally. If the URL passes these, 
   our FastAPI backend uses 'asyncio' to query Google Safe Browsing, VirusTotal, 
   and AbuseIPDB in parallel. By running these network requests concurrently 
   rather than sequentially, we drop scan wait times from 8 seconds to under 1.5 
   seconds, delivering a seamless security response to the client dashboard."

SLIDE 7: Module 3 - Adaptive Risk Engine & Time-Decay Math
--------------------------------------------------------------------------------
[Slide Title]: Module 3: Dynamic Adaptive Risk Engine
[Slide Bullets]:
  * Dynamic Accumulation: Security incidents increment a user's risk score based
    on severity weights (e.g., failed logins +15, blocked port scan +40).
  * Exponential Time-Decay Formula:
      Risk(t) = Initial_Risk * e^(-lambda * t)
  * Reward-Based Decay: Risk naturally and safely decays as time (t) passes 
    without new suspicious events.
  * Real-Time Alerts: Scores exceeding 70 trigger admin notification warning 
    emails generated dynamically via the Resend API.

[Visual Recommendations]:
  * A math highlight card displaying the formula: Risk(t) = R_0 * e^(-λt) 
  * A graph line showing a sharp spike in risk (an incident) followed by a 
    smooth exponential curve decaying back to zero over time.

[Speaker Notes]:
  "Module 3 is the core brain of SmartSec: the Adaptive Risk Engine. Rather 
   than treating all historical security warnings identically, we apply an 
   exponential mathematical decay model. The risk score spikes instantly when an 
   intrusion is detected, but decays exponentially over time using the formula 
   Risk = Initial Risk multiplied by e to the negative lambda-t. If a user 
   exhibits good behavior, their risk returns to normal. However, if the risk 
   crosses 70, the backend calls the Resend API to dispatch an urgent HTML warning 
   email to system administrators."

SLIDE 9: Relational Database Schema & Schema Integrity
--------------------------------------------------------------------------------
[Slide Title]: Secure Relational Database Architecture
[Slide Bullets]:
  * Profiles: Stores usernames, email notification preferences, and dynamic risk ratings.
  * Login History: Audits session tokens, user agents, IP addresses, and login success flags.
  * Phishing Scans: Logs scanned domains, static heuristic results, external API verdicts,
    and the requesting user ID.
  * Notifications: Houses broadcast alerts, security trigger levels, and is_read states.
  * Cryptographic Authentication: JWT tokens containing custom claims, served through HTTP-only,
    SameSite cookies to neutralize Cross-Site Scripting (XSS) threats.

[Visual Recommendations]:
  * An Entity-Relationship (ER) diagram mapping:
    - [PROFILES] (1) <----> (N) [LOGIN_HISTORY]
    - [PROFILES] (1) <----> (N) [PHISHING_SCANS]
    - [PROFILES] (1) <----> (N) [NOTIFICATIONS]

[Speaker Notes]:
  "For storage, we structured a highly normalized database inside Supabase.
   We maintain four main tables: Profiles, Login History, Phishing Scans, and 
   Notifications. To secure communication, we do not store raw tokens in the browser's 
   local storage. Instead, our backend issues cryptographically signed JSON Web 
   Tokens saved inside HTTP-only, SameSite cookies. This layout blocks client-side 
   JavaScript from reading the session, effectively neutralizing Cross-Site Scripting."

SLIDE 11: Real-World Incident Handling Simulation
--------------------------------------------------------------------------------
[Slide Title]: Threat Event Lifecycle Simulation
[Slide Bullets]:
  * 1. Threat Inception: A malicious sweep or failed login attempts are simulated.
  * 2. Anomaly Isolated: The unsupervised Isolation Forest tags the traffic vector.
  * 3. Math Recalculation: Dynamic Risk Engine spikes the target score above 70.
  * 4. Notification & Email Dispatch: Database records are updated, and the backend
    sends an administrative notification via the Resend API.
  * 5. Threat Decays: Safe operations resume, and the risk score slowly and safely
    decays over time back to safe limits.

[Visual Recommendations]:
  * A clean circular diagram illustrating the continuous loop of Threat -> 
    Detection -> Risk Spike -> Notification -> Decay -> Normal state.

[Speaker Notes]:
  "Let's trace how SmartSec works in a real-world scenario. If a port sweep is 
   attempted:
   1. The transaction is instantly processed by the Isolation Forest.
   2. Recognizing the outlier signature, it flags it as an anomaly.
   3. The Risk Engine adds +40 to the user's score, pushing it to 85.
   4. The Resend service fires an email warning to the security administrator.
   5. As the administrator secures the system and no more threats occur, the score 
      safely decays back to 0. This lifecycle represents proactive security."
